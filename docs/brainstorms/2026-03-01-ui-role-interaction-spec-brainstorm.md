# 基于 Harness Engineering 终极目标的 UI 交互规格

## What We're Building

为 5 个角色设计完整的 UI 交互体系，每个角色有明确的页面集、操作权限、数据边界和不可违反的设计原则。UI 是 Harness 7 层的"人机界面"——让人类能驱动、观测、干预、进化这个 Agent 系统。

## 设计铁律

每个角色有一条**绝对不可违反**的设计原则，直接对应 Harness 层：

| 角色 | 铁律 | 对应 Harness 层 | 工程强制方式 |
|------|------|-----------------|-------------|
| Operator | 拒绝 Agent 建议时**必须选原因** | Layer 7 进化层 | reject API 必须携带 `rejectionReason`，前端无"跳过"选项 |
| Tenant Admin | Agent 授权中心**不能跳过** | Layer 2 执行层 | 新 Agent 能力默认关闭，必须在授权中心显式开启 |
| System Admin | **看不到**单一租户业务数据 | Layer 1 基础层 | API 层强制：admin 端点只返回聚合/脱敏数据 |
| Supplier | 前端+后端**双重隔离**售价利润 | Layer 1 基础层 | Supplier 视图无 price/margin 字段，API 不返回 |
| Viewer | **零操作按钮**，含 AI 效能数据 | Layer 6 观测层 | 前端无任何 POST/PUT/DELETE 调用，纯 GET |

## 角色 1：Operator（租户运营人员）

### 核心使命
> 日常与 Agent 协作的人。Agent 的建议在这里被执行、修正或拒绝。每一次交互都是进化层的燃料。

### Mobile 优先级：🏆 最高
随时处理 Agent 紧急待办，左滑右滑操作。

### 页面清单

**交互规则：**
- 每条待办必须展示：Agent 建议 + 置信度 + 依据样本量
- "执行"：一键确认，Agent 立即执行
- "修改"：展开参数面板，Operator 可调整具体数值后执行
- "拒绝"：**必须选择拒绝原因**（枚举 + 自由文本），这是进化层的核心燃料
- 拒绝原因枚举：`市场判断不同` / `时机不对` / `数据不准` / `风险太高` / `其他`
- Mobile：左滑 = 执行，右滑 = 拒绝（弹出原因选择）

**Harness 层映射：**
- Layer 2 执行层：Agent 待办的状态流转
- Layer 6 观测层：置信度 + 依据展示
- Layer 7 进化层：拒绝原因收集

## 角色 2：Tenant Admin（租户管理员 / 卖家公司负责人）

### 核心使命
> 决定 Agent 能做什么、不能做什么。管理团队成员权限。对 Agent 的"宪法"负最终责任。

### Mobile 优先级：⚡ 高
随时审批 Agent 授权决策。

- Mobile：开关式交互，关键参数可快速调整

**Harness 层映射：**
- Layer 2 执行层：Agent 能力边界定义
